### Architectural mismatch — the real correction

`snobol4.l` returns tokens **directly to the Bison-generated `yylex`** in a
single pass.  Each Flex rule says `return T_FOO;` and Bison pulls them via
the standard reentrant `yylex` interface.  No accumulation, no buffer.

Today's `snocone.l` instead **accumulates tokens into a `ScTokenBuf`**
inside `snocone_lex2()`, and my draft `snocone.y` consumed that buffer via
a `sc_yylex` thunk that translated `ScKind` enum values to Bison's enum.
This was a leftover from the LS-3.c standalone token-stream tests
(`test_snocone_lex2.c`) that Lon left in place; it's a two-pass design and
the wrong shape for the integrated parser.

**Correction for next session:** make the Snocone Flex+Bison pair behave
exactly like the SNOBOL4 pair —

1. **`snocone.l` rewrite (LS-3.f, new sub-step):**  Each rule `return T_*;`
   directly.  Drop the `emit_simple()` accumulator, drop `ScTokenBuf`, drop
   `snocone_lex2()` public API.  Use `%option reentrant noyywrap` matching
   `snobol4.l`.  Token kind values come from `snocone.tab.h` (generated by
   Bison), not from `snocone_lex2.h`.  YYSTYPE token-value union is the one
   `snocone.y` declares.

2. **`snocone.y` rewrite (LS-4 proper):**  Use the standard Flex/Bison
   single-pass plumbing.  Match `snobol4.y`'s shape: `%define api.prefix
   {sc_}`, `%define api.pure full`, `%parse-param { void *yyparse_param }`,
   no manual `sc_yylex` thunk — Bison generates the call to the lexer.  IR
   construction in semantic actions (the part of my draft that *is* right)
   ports unchanged.

3. **Test bridge for LS-1.b (the 31-test token-stream gate):**  Move
   `test_snocone_lex2.c` from production code to `test/` so its dependency
   on `ScTokenBuf` doesn't constrain the production lexer.  The test can
   continue to drive Flex via `yylex_init` + token-streaming and accumulate
   for assertions — that's a test-harness concern, not a production
   architecture.  Or simpler: regenerate the LS-1.b acceptance run from a
   wrapper that calls `yylex` in a loop.

4. **`snocone_control.c` removal (LS-4):**  Once the new `.y` handles
   control flow in grammar productions, delete `snocone_control.c`,
   `snocone_parse.c`, `snocone_lower.c`, `snocone_lex.c` (old hand-rolled),
   and update `src/Makefile` to compile `snocone.tab.c` + `snocone.lex.c`
   only (matching `FRONTEND_SNO`'s shape).

5. **Driver collapse (LS-4):**  `snocone_compile()` becomes a 5-line wrapper
   — set up Flex scanner state, call `sc_parse()`, return the `Program*`.
   Match `sno_parse_string()` in `snobol4.y` line 316.

## Architectural lesson recorded

The two-pass design (lex into buffer, then parse from buffer) felt natural
for testing in LS-3.c but creates an unnecessary impedance mismatch at
LS-4 — a thunk that translates one enum to another, plus a `ScParseState`
that owns a buffer cursor instead of letting Flex own its scanner state.
**Single-pass Flex→Bison via the standard `yylex` API is one4all's
established convention** (snobol4, raku, rebus all do this).  Snocone
should match.
